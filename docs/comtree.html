Eksempel på commit tre

Figur X. Et bilde av et skriftlig dokument med metadata og en agenda som innhold (brukt på samling 4. Sept. 2023).

La oss bruke dette dokumentet som et utgangspunkt for en repository på Github. Jeg oppretter en repository (ved å bruke knappen "New"; se Figur X) i brukerkontoen uia-worker med navn is-114. Som dere husker fra forelesning, så brukte vi UiA som metoforen for en sertifiserende autoritet. I den elektroniske "verden" skal vi anta at Github er en slik autoritet som vi alle stoler på. Vi bruker de samme navn (identifikatorer) på eier og boks (repository) i den elektroniske "verden", som vi brukte i den fysiske verden.

Figur X. Skjermdump demonstrerer knappen "New", som kan brukes for å lage en ny repository (skjermdump laget 5. Sept. 2023).

Det er flere valg i den elektroniske "verden", som ikke fantes i den fysiske verden (se Figur X). Det er to type repository-er på Github, - Public (alle med på Internett kan se innholdet i repository-en, men eieren kan bestemme hvem kan skrive til repository-en) og Private (eieren kan bestemme hvem som kan se innholdet og skrive til repository-en). I dette tilfelle velger jeg Public fordi at jeg ønsker at et stort antall utviklere kan se innholdet i denne repository-en. Noen applikasjoner tilbyr også valget "Unlisted", som gjør at alle som kjenner til lenken kan se repository, men den er ikke listet ut i brukerens profil.

Figur X. Skjermdump demonstrerer knappen valg gjort før man lager en ny repository (skjermdump laget 5. Sept. 2023).

Jeg ønsker også at programmet på Github skal lage en fil med navn README.md i det repository-en blir opprettet. Det er en "de facto"-standard (uformell lov) at hver repository skal ha en fil som beskriver prosjektet som en helhet. Jeg kunne skrive noe som

Denne repositorien inneholder oppgaver for emne IS-114 ved Universitetet i Agder. Repositorien kan brukes for å lære å bruke Git og Github.
Foreløpig ingen dyptgående beskrivelse, men den kan utvides senere.

Foreløpig lar vi valget "Add .gitignore" ligge (la stå "None"). Dette valget kommer vi tilbake til senere.

Jeg velger derimot en lisens for mitt arbeid, - "Creative commons Zero v1.0 Universal license". Denne lisensen ønsker jeg å bruke fordi at den er designet for å maksimalt fremme deling, gjenbruk og adopsjon av kreativt arbeid (åndsverk) uten begrensninger og tillate andre til å bruke arbeidet for ethvert formål, inkludert kommersiell bruk, uten å søke tillatelse eller gi attribusjon til den opprinnelige skaperen. Det finnes flere type lisenser, som brukes for å fremme åpen tilgang til kunnskap og dere vil bli kjent med dem i løpet av studiet.

Da er alt klart for å skape en ny repository!

Figur X. Viser en nylig skapt repository på Github med navn "IS-114 (skjermdump laget 5. Sept. 2023).

Som vi kan se i Figur X, har Github applikasjonen skapt et grensesnitt for oss, med mange funksjoner. Dere kan see "Issues" (vi nevnte de på samlingen) i den horisontale menyen nesten på toppen av siden. Dere kan også se "Fork" knappen neste øverst til høyre, som gir mulighet å kopiere repository-en til sin egen konto og eventuelt videreutvikle den videre uten direkte tilknytting til den opprinnelige (videre kalt originale og i git-terminologien kalt origin) repository-en. Under "Settings" finnes det flere valg, blant annet for å gi samarbeidspartnere rettigheten til å skrive til (gjøre git push) den originale repository-en. For å lage en "Personal Access Token", må dere bruke "Settings" for deres konto og ikke "Settings" for repository-en. Dere kan gjerne selv utforske flere funksjoner på siden.

Før vi forlater webgrensesnittet fra Github, skal vi kopiere lenken som skal gi oss mulighet til å klone repository-en på vår lokale datamaskin ved å skrive kommandoen git clone på kommandolinje (Terminal, GitBash, cmd eller PowerShell). Se Figur X, som illustrerer dette. For å kunne skrive tilbake til repository, er det nødvendig å konfigurere autentisering, siden default er det kun eieren av repository-en som kan skrive til den på Github. Hvis du har laget en "Personal Access Token" og installert Git på din datamaskin korrekt, så kan du velge metoden "HTTPS". Vi forklarer de andre metodene senere.

Figur X. Viser valg for kloning i webgrensesnittet på Github (skjermdump laget 5. Sept. 2023)

Bilde ovenfor har en dårlig oppløsning, men lenken til repositorien er https://github.com/uia-worker/is-114.git. Vi kan si litt om hvordan dette navnet (som er en global identifikator på WWW) er konstruert. Først er det protokoll, som brukes for å kommunisere mellom to noder (les datamaskiner) på Internett, - https://, som står for HyperText Transfer Protocol. Så kommer det "sertifiserings autoritet" eller "servernavn", som i vårt tilfelle er github.com. Det finnes flere webapplikasjoner som tilbyr lignende tjenester som Github, - Bitbucket, Gitlab, sourcehat

La oss analysere linjene fra output på kommandolinjen vist i Figur X.

  • janisg@UIAA223034AS is-114 % git log er en såkalt prompt sammen med et kommando og dets subkommando. Fra latin promptus, som betyr beredt. I denne konteksten kan man tolke det at brukeren får en tekstuell hint og en markør, som inviterer til handling (noen ganger kaller man det også affordance, som kan oversettes til norsk som råd). I dette eksemplet er output generert på en MacBook Pro datamaskin og derfor er annerledes enn, for eksempel, i GitBash. janisg@UIAA223034AS indikerer brukernavn for bruker som bruker terminalen sammen med en kodeord, som indikerer navn til datamaskinen (blir valgt når datamaskinen blir konfigurert for første gang). Tegnet "%" brukes har som et prompt-tegn, dvs. den skal signalisere at etter det kan bruker skrive inn sine kommandoer. Før man har skrevet inn noe, skal det også vises en markør (kan være en stedsholder eller noe som fremstår som blinkende). I zsh (Wikipedia Contributors, 2023a) og i macOS (Wikipedia Contributors, 2023b) kan man modifisere promten ved å skrive inn spesifikke kommandoer i en spesifikk konfigurasjonsfil med navn ~/.zshrc, hvor ~ representerer brukerens hjemmemappen (Williams, 2022).
    git log er kommandoen og subkommandoen, som jeg skrev på kommandolinje og som vi kaller for input. Mer informasjon om git log kan man finne ved å skrive inn uttrykket git log --help (den består av hovedkommando, subkommando og en en såkalt alternativ (på engelsk option).
  • commit ea38683d0e38e99c1b04b943f8b7c2092ef4b32a (HEAD -> main, origin/main, origin/HEAD) er linje fra Git og er den første linje i output. Den viser en id for commit, som er en SHA-1 hash. Kort forklart er det en ID, som består av tall og bokstaver (en streng), og er generert på en møte (med en algoritme), som gjør at sjansen for å generere to like slike strenger innen en repository er tilnærmet null. I vår "commit-tre" skal vi kun notere de første 7 symboler fra denne strengen. I parentes så er det ekstra informasjon om tilstanden til Git-databasen. Git bruker et begrep HEAD, som man kan forestille seg som en slags markør som markerer den gjeldende grenen (på engelsk current branch), som i dette tilfelle er main. Det indikeres også at den lokale og den originale grenen er synkronisert origin/main, og at den lokale og den originale HEAD origin/HAED, peker på toppen av sine respektive grener.
  • Denne linjen bør være selvforklarende. Husk at navnet og e-postadressen er den som ble definert med 
    
        git config --global user.name "(ditt navn)"
        git confit --global user.email "(din e-postadresse)"
  • Dato eller tidsstempel (fra den engelske timestamp). Definisjon av formatet er en innstiling i Git men kan være avhengig av innstillingen for operativsystemet på din datamaskin. Vi går ikke inn på det her.
  • Etter en tom linje kommer det en kommentar for hva commit-en dreier seg om. Husk at en commit kan relateres til endringer i flere enn en fil, så beskrivelsen burde gjenspeile alle endringene så kortfattet som mulig.

Før vi går videre, kan vi introdusere et nyttig kommando-uttryk git status, som vi vil bruke mye.

Figur X. Viser output fra kommando-uttrykket git status

Jeg bruker ordet kommando-uttrykk for å illustrere at det inneholder en hovedkommando git og en subkommando status. Dette kan være forvirrende, siden vi brukte også et ord alternativ. Et uttrykk på kommandolinje kan ha mange former. De kan også være forskjellige fra forskjellige operativsystemer og i forskjellige terminaler. Vi definerer følgende (men vær oppmerksom på at denne representerer ikke noen form for standardterminologi):

  • et kommando er et ord som man skriver på kommandolinje som kaller opp en applikasjon (som git, ls, pwd, men her kan også begrepene være tvetydige, siden mange kaller disse for verktøy eller utilities på engelsk)
  • et subkommando er et type alternativ, som representerer en større funksjonalitet i seg selv (for eksempel git add, go build, doctl apps osv.); noen applikasjoner introduserer også subsubkommandoer
  • et alternativ indikerer at applikasjon oppfører seg vesentlig annerledes og som vanligvis starter med tegn - eller --
  • Et flagg, som enten slår en funksjonalitet av eller på.

Jeg gir filen etternavn .txt (tekst), siden agenda ble ikke skrevet på et spesifikt programmeringsspråk, men på et naturlig språk.
