Git

Git er versjonskontrollsystem (et dataprogram), som oppsto av nødvendighet for sikker og effektiv samarbeidsmetode for utviklere av programvare, spesielt relatert til åpen kildekode programvare. Utviklingen av Git ble startet av Linus Thorvalds (The Linux Foundation, 2015) i 2005, og har blitt utviklet i åpen kildekode felleskapet siden.

Åpen kildekode programvare prosjekter ofte har sporadiske bidrag fra et stort antall utviklere (kodere eller programmerere) og det fører til et behov for et system, som begrenser muligheter for at feil blir lagt inn i programvaren, spesielt i den gjeldende versjonen, som kalles for produksjonsversjon. I slike prosjekter opererer man også ofte med en annen versjon av programvare, som kalles for utviklingsversjon (på engelsk "development versjon"). Regler for å legge inn endringer i koden er mindre strenge for utviklingsversjonen enn for produksjonsversjonen og utviklingsversjonen brukes for å teste ut programvaren etter at feil har blitt rettet (på engelsk kaller man det "hotfix" eller "bugfix") eller når nye funksjoner (eller brukstilfeller, som kan inkludere flere funksjoner og eventuelt moduler) er lagt til.

I et programmeringsmiljø bruker man ofte begrepet fil om en mengde data lagret som en enhet (fra et symbol for begynnelse til et symnbol for slutt) og gitt et [unikt] navn. I dagliglivet bruker man ofte begrepet dokument. Et dokument er en av artefaktene (Det Norske Akademis Ordbok, 2023a), som homo sapiens bruker for å bevare sine mentale modeller og for å kommunisere (f. eks. definere relasjoner med hverandre). Om det er et manus for en roman, besvarelse av en skoleoppgave, vitenskapelig artikkel skrevet på et spesifikt "naturlig" språk (f. eks. norsk, engelsk, urdu), juridisk kontrakt om et kjøp av eiendom, eller et program skrevet på et programmeringsspråk (f. eks. Pyret, Python, JavaScript), så avgrenser homo sapiens det ofte med et begrep dokument eller, som nevnt tidligere, - fil (fra ca. 1950 om informasjon lagret på en datamaskin som én enhet med ett navn). I vår kontekst (IS-114) definerer vi dokument som elektronisk tekst som kan lagres, redigeres og gjenfinnes (Det Norske Akademis Ordbok, 2023b), men vi vil hovedsakelig bruke begrepet fil om dokumenter som inneholder kildekode. Git kan da betraktes som et program som lagrer endringene i en mengde filer, som hører til en større enhet, som vi kaller for applikasjon og som er distribuert på flere datamaskiner. Git hjelper å synkronisere endringene i filene fra flere utviklere, som ofte befinner seg på forskjellige [fysiske] lokasjoner i et større datanettverk (f. eks. Internett). Man kan også betrakte Git, som en distribuert databasesystem. Vi skal etterhvert finne ut at i Git applikasjonen brukes det en egen mappe (med navn .git) på filsystemene til hver av datamaskinene, som inneholder data om endringene av den applikasjonen som utviklere jobber med. De forskjellige kommandoene (som vi kan også kalle programmer), gir muligheter til å synkronisere alle databasene over et nettverk og registrere alle endringene fra alle utviklere.

Git har dannet et grunnlag for verdens største kodebase, som er i hovedsak er åpen for alle, som har tilgang til Internett. Selv om mange kommandoer, som hører til Git applikasjonen, kan fremstå som vanskelige å forstå med en gang, er hovedprinsippet om å gjøre samarbeid på den samme kodebasen sikker og produktiv, forholdsvis enkelt. Når det gjelder forståelsen og anvendelsen, så er hovedbudskapet her det samme som mange andre steder, - øvelse gjør mester.

I en del av punktene i oppgaven, er det lagt inn noen spørsmål, som er ment som hjelp til refleksjon om det, som dere praktisk gjennomfører på deres datamaskiner.

Prosjektoblig 03

Individuell del

  1. Installere et "minimalistisk" tekstredigeringsverktøy, som Sublime Text (for macOS, Linux og MS Windows) eller Notepad++ (kun for MS Windows). Hvorfor anbefales det ikke å bruke IDE (Integrated Development Environment) som Visual Studio Code eller IntelliJ hvis man er nybegynner i systemutvikling?
  2. Installere Git på egen datamaskin. Hva er Git og hvorfor trenger man å bruke det? Hvorfor er det ikke anbefalt å installere en GUI (Graphical User Interface) klient applikasjon for Git?
  3. Gjøre seg kjent med grunnleggende bruk av applikasjonen Terminal i macOS eller Linux (Ubuntu 22.04), og GitBash i MS Windows operativsystemet. Hvilke grunnleggende kommandoer trenges for å kunne se filene, som er lagret på egen datamaskin på kommandolinjen? Hvordan finner jeg de samme filene i Terminal/GitBash som jeg ser i applikasjonen med grafisk grensesnitt (explorer?) for fremvisning av mappene hvor filene er lagret (filsystemet)? Hvorfor er det anbefalt å lære å bruke Git-kommandoer fra kommandolinjen?
  4. Lage egen konto på github.com og logge inn. Hvorfor trenger man en konto på Github?
  5. Lage en "personal access token" (selektere alt for "repo") for kontoen. Hvorfor trenger man en slik "token"?
  6. Lage en offentlig (ikke privat) repository (velge et meningsbærende navn for repository, velge en lisens, velge standard README-fil) på Github. Hva er en repository? Hvorfor skal navn være meningsbærende? Hvorfor velge lisens? Hva betyr README-fil og hvorfor er det anbefalt å ha det i hver repository? Hva betyr det at repository-en er på Github? Hva betyr det at repository har en gren (på engelsk branch) som heter main?
  7. Klone (kopiere) repository-en fra Github på sin egen datamaskin. Hvorfor skal man kopiere repository-en fra Github til egen datamaskin? Hvordan velger man et egnet sted å kopiere repository-en til på egen datamaskin? Hvilke filer og eventuelt mapper som blir kopiert fra Github til egen datamaskin med kommandoet for kloning?
  8. Åpne en tekstredigeringsverktøy på egen datamaskin, åpne mappen hvor repository-en fra Github ble klonet (lagret i), velge filen README.md, skrive inn navn og versjon for operativsystemet på egen datamaskin på en ny linje og lagre endringene. Er det alltid nødvendig å åpne egen applikasjon for tekstredigering for å kunne gjøre endringer i filene? Kunne jeg endret andre type filer på samme måte, som, for eksempel, bilde- eller videofiler? Begrunne svar.
  9. Bruk egnede Git-kommandoer på kommandolinjen for å
    • sjekke status for endringene i den gjeldende mappen
    • registrere den endrede filen (ny versjon av filen og generelt endring i den gjeldende mappen) for innlegging i den lokale repository-en ("legge filen på scenen" eller på engelsk "add to staging area"),
    • forpliktende (på engelsk "commit") legge inn den nye versjonen av filen (endringene av filen og generelt i den gjeldende mappen) i repository-en,
    • kopiere eller overføre (på engelsk "push") endringene til en annen instans (som kan ha samme versjon som ved siste synkronisering eller en endret versjon) av den samme repository-en.
    Hvilken er gjeldende mappe på kommandolinjen? Hvorfor trenger man flere operasjoner for å registrere endringene i Git versjonskontrollsystemet? Kan man klone og kopiere endringene tilbake for en repository man ikke selv eier (dvs. repository-er som andre har laget ved hjelp av sine kontoer på Github)? Hvorfor er det nyttig å kopiere endringene gjort i en fil på egen datamaskin til Github?
  10. I nettleseren, i grensesnittet for github.com, og pålogget med din konto, gjør endringer i README.md filen. Legg inn brukernavnet for din github-konto på en ny linje i README-filen. Lagre filen i Github.
  11. Gå til tekstredigeringsverktøyet på din lokale datamaskin igjen og legg inn en ny linje i README.md med versjonsnummer til Git applikasjonen som du har installert på din egen datamaskin (for eksempel "git version 2.30.1 (Apple Git-130)". Lagre endringene. Prøv å registrere den endrede versjonen av README.md, legg inn endringene i den lokale versjonen av repository-en og sende/pushe endringene til Github. Hvilke respons genererer "push"-kommandoen? Hva er grunnen til en slik respons?
  12. Prøv å finne en løsning for å kunne ta vare både på de endringene som ble gjort i Github og de som ble gjort i den lokale repository-en.
  13. Konfigurer "Github pages" for din repository i Github fra grenen main og i mappen /docs. Hva er funksjonen til "Github pages"?
  14. I din lokale repository-en lag en mappe med navn docs og i denne mappen lag en fil med navn index.html. Hvorfor gjør man endringene i den lokale repository-en og ikke i Github direkte?
  15. I denne filen legg inn følgende kode: 
    
<html>
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>(navn [mellomnavn]* etternavn)</title>
<</head>
<body>
  <h1>Min IS-114 portfolio</h1>
  <p>Kort beskrivelse av portfolio (oppgaveløsninger, dagbok, notater osv.) ... </p>
</body>
</html>
    Hvilken type kode er denne? Hvordan kan denne koden bli utført?
  16. Bruk git-kommandoer for å lagre endringene i Github. Hva betyr det at en webside er tilgjengelig på Internett?
  17. Gjør endringer (legg inn ditt navn i tittelen og lag en kort beskrivelse av din portfolio så langt) i filen index.html, lagre endringene lokalt og sende endringene til Github igjen. Hvorfor er en slik fremgangsmåte nyttig ved utvikling av websider?

Gruppedel

  1. Velg hvem av gruppemedlemmene, som skal eie HOVEDREPOSITORY.
  2. Eieren av HOVEDREPOSITORY lager en ny offentlig (ikke privat) repository med et navn som hele gruppen velger.
  3. Eieren av HOVEDREPOSITORY inviterer alle de andre gruppemedlemmene som samarbeidspartnere i denne repositorien.
  4. Bruk erfaring fra den individuelle delen og lag "Github pages" med index.html i mappen /docs i HOVEDREPOSITORY.
  5. Alle gruppemedlemmene kloner HOVEDREPOSITORY til sin lokale datamaskin.
  6. Alle gruppemedlemmene lager en gren med navn feature-(initialer) (hvis noen av gruppemedlemmene har like initialer, så må dere finne på en måte å skille mellom dem).
  7. I denne grenen endre index.html ved å legge inn et avsnitt med eget visitkort (kanskje lurt å unngå å legge ut mobilnummer, siden såkalte "crawlere" søker opp alle offentlig tilgengelige sider på WWW og "skraper" disse for informasjon som kan brukes i spamming/markedsføring).
  8. Legg inn en "Pull request" for din gren i HOVEDREPOSITORY.
  9. Alle sammen følger med på når eieren til HOVEDREPOSITORY godtar og legger inn alle endringene i grenen main.

Hva skal evalueres?

Vise deres læringsassitent den individuelle repository-en og websiden basert på den. Vise frem gruppe repository-en og websiden basert på den. Vise frem status for begge repository-ene på kommandolinje lokalt på sin datamaskin. Websidene må vises frem korrekt, dvs. ha gyldig HTML-kode. README.md i den lokale repositorien må inneholde 3 linjer (1: operativsystemets navn og versjon, 2: brukenavnet for Github-konto, 3: versjonsidentifikator til Git). "Github pages" må være konfigurert korrekt både i de individuelle og gruppe repository-en. HOVEDREPOSITORY må inneholde minst en "Pull request" fra hver av gruppemedlemmene. Bidragene fra gruppemedlemmene må være lagt inn i index.html filen og vises frem på websiden (gjennom "Github pages"). I tilfelle det har oppstått problemer med gjennomføring av noen av punktene i oppgaven, skal studenten eller gruppen forklare sin fremgangsmåte og beskrive problemet for LA-en. Dere må også være forberedt på å svare på hver av spørsmålene, som står i oppgaveteksten (LA-en vil velge hvilke spørsmål som blir stilt). Godkjent/Ikke godkjent, men det blir gitt flere forsøk på å få innleveringen godkjent.

Appendix II: Installasjon

Installasjon av Git varierer avhengig av operativsystemet. Våre studenter bruker hovedsakelig tre forskjellige operativsystemer, - MS Windows, mocOS og Linux. De fleste bruker MS Windows og macOS, mens veldig få bruker Linux (oftest Ubuntu distribusjonen). Under [manuell] installasjon av all programvare på egen datamaskin, må man regne med at utfordringer kan oppstå. Selv om noen datamaskiner kan være fysisk tilnærmet like (dvs. har den samme maskinvaren), er aldri like når det gjelder programvaren som er installert på dem. Derfor ingen guider, som man finner på Internett, vil garantere en korrekt installasjon. Du som bruker må regne med å bruke tid på å finne ut av problemene når de oppstår og gjennomføre tester for å forsikre deg at programvaren funksjonener som forventet.

Her er de vanligste spørsmål, som kommer opp i form av dialogbokser under installasjonen av Git:

  • Velge Git GUI - kan beholde denne, siden den vil vise noen grafiske symboler i MS Windows Explorer, når filsystemet blir fremvist i det grafiske grensesnittet til MS Windows.
  • Velge Git LFS - LFS står for Large File Storage og er anbefalt hvis man ønsker å lagre store [binære] filer, som bilder eller videoer i sin Git repository. Git var ikke designet for å kontrollere versjoner på binære filer, så dette valget har kommet inn senere, siden det er mulig å få forholdsvis stor gratis lagringsplass gjennom webapplikasjoner som Github og Gitlab, for eksempel. For oppgavene i IS-114 skal det ikke være behov for superstore filer, men siden dette valget er default og har ingen store implikasjoner for vanlig bruk, så kan man la den være selektert.
  • Valg av default (forhåndsvalgt alternativ i et dataprogram) tekstredigeringsverktøy - dette er et viktig valg siden noen av Git-kommandoene vil prøve automatisk å starte en applikasjon for tekstredigering. Grunnen til det er at alle forpliktende registreringer av endringer i den gjeldende mappen skal ha en kommentar om hva endringene dreier seg om. Det regnes som dårlig praksis blant utviklere, hvis man ikke er nøye med å kommentere alle forpliktende ("commit") registreringer i repository-en. For eksempel, hvis man har lagt en fil "på scenen" for endringer og utfører kommando 
    git commit
    i den gjeldende mappen, vil default tekstredigeringsverktøy forsøke å starte opp. Derfor er det viktig at før man installerer Git, har man allerede installert et egnet tekstredigeringsverktøy. Anbefalt verktøy for IS-114 er Sublime Text og Notepad++. OBS! Ofte når man jobber i kommandolinje, ønsker man ikke avbrytelser som at en ny applikasjon starter opp. Tekstredigeringsverktøy er blitt store applikasjoner og en slik oppstart kan ta tid. Derfor kan man velge å skrive kommentaren direkte i kommandolinje ved å bruke ekstravalget -m, som står for message (husk at meldingen må skrives mellom doble anførselstegn): 
    git commit -m "Endret fontstørrelsen for brødteksten"
    Et alternativ er også å beholde default tekstbehandler, som vanligvis er vim. Da må man lære noen grunnleggende kommandoer i vim. Hvis man allerede har erfaring med systemutvikling/programmering, kan man bruke IDE-er som Visual Studio eller IntelliJ. IDE står for Integrated Development Environment. Studentene kan ikke regne med å få hjelp med problemer som eventuelt oppstår i forbindelse med bruken av IDE-er.
  • Velge applikasjon som utføres for å sørge for SSH - SSH står for Secure [Socket] Shell og garanterer ende til ende kryptering for kommunikasjon mellom den lokale kommandolinjen og den eksterne tjeneren (i vårt tilfelle en tjener som Github foretaket administrerer). OpenSSH er her det beste valget, som også skal være default.
  • Velge PATH - PATH er et miljøvariabel og er en samling av stier som peker til steder hvor utførbare programmer ligger. Stien som legges inn i PATH for Git, er en sti (absolutt navn til en mappe) hvor den utførbare filen til Git ligger. Valget her bør være "Git from command line and also from 3rd-party software". Da vil man kunne utføre Git-kommandoer både i GitBash, cmd og PowerShell under MS Windows operativsystemet.
  • Velge initiell gren - det engelske ordet er "branch"; velg main som standardnavn til den initielle "branch"-en (master ble forkastet på grunn av politiske grunner).
  • Velg "chechout style" (utsjekkingsstil) - velg "Checkout Windows-style, commit Unix-style line endings". Dette valget er grunnet forskjeller på hvilke binære symboler som indikerer linjeskift i tekstfiler. Hvis en utvikler jobber på MS Windows plattform og en annen på macOS plattform, så må linjeendingene konverteres, hvis tekstredigeringsverktøyene på begge plattformene skal vise filen korrekt.
  • Velg terminalemulator - la den være default. Vi kommer senere tilbake til hva det er, men du er velkommen til å finne det ut selv.
  • Velg default adferd ved git pull - denne kommandoen synkroniserer repositorien på en annen datamaskin med din lokale repositorien (for eksempel, når den versjonen som du sjekket ut fra en annen datamaskin, er blitt oppdatert av en annen og sammenfaller ikke lenger med din versjon). La denne står default, dvs. at man gjør en "fast-forward" eller "merge" (kommer tilbake til hva det betyr etterhvert, men du er velkommen til å finne det ut selv).
  • Velge "Git Credential Manager" - la den være default. Den skal administrerer dine personlige "tokens", slik at du ikke trenger å skrive inn brukernavn og passord hver gang du skal oppdatere en ekstern, tidligere utsjekket versjon av repository. Husk å utføre disse to kommandoene før du begynner å bruke b>Git (bruk brukernavn og e-postadressen som du brukte når du opprettet en konto på Github): 
    
        git config --global user.name "(ditt navn)"
        git confit --global user.email "(din e-postadresse)"
  • Andre valgmuligheter kan man la stå med default.
