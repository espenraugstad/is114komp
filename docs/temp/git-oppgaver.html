Prosjektoblig 03

Individuell del

  1. Installere et "minimalistisk" tekstredigeringsverktøy, som Sublime Text (for macOS, Linux og MS Windows) eller Notepad++ (kun for MS Windows). Hvorfor anbefales det ikke å bruke IDE (Integrated Development Environment) som Visual Studio Code eller IntelliJ hvis man er nybegynner i systemutvikling?
  2. Installere Git på egen datamaskin. Hva er Git og hvorfor trenger man å bruke det? Hvorfor er det ikke anbefalt å installere en GUI (Graphical User Interface) klient applikasjon for Git?
  3. Gjøre seg kjent med grunnleggende bruk av applikasjonen Terminal i macOS eller Linux (Ubuntu 22.04), og GitBash i MS Windows operativsystemet. Hvilke grunnleggende kommandoer trenges for å kunne se filene, som er lagret på egen datamaskin på kommandolinjen? Hvordan finner jeg de samme filene i Terminal/GitBash som jeg ser i applikasjonen med grafisk grensesnitt (explorer?) for fremvisning av mappene hvor filene er lagret (filsystemet)? Hvorfor er det anbefalt å lære å bruke Git-kommandoer fra kommandolinjen?
  4. Lage egen konto på github.com og logge inn. Hvorfor trenger man en konto på Github?
  5. Lage en "personal access token" (selektere alt for "repo") for kontoen. Hvorfor trenger man en slik "token"?
  6. Lage en offentlig (ikke privat) repository (velge et meningsbærende navn for repository, velge en lisens, velge standard README-fil) på Github. Hva er en repository? Hvorfor skal navn være meningsbærende? Hvorfor velge lisens? Hva betyr README-fil og hvorfor er det anbefalt å ha det i hver repository? Hva betyr det at repository-en er på Github? Hva betyr det at repository har en gren (på engelsk branch) som heter main?
  7. Klone (kopiere) repository-en fra Github på sin egen datamaskin. Hvorfor skal man kopiere repository-en fra Github til egen datamaskin? Hvordan velger man et egnet sted å kopiere repository-en til på egen datamaskin? Hvilke filer og eventuelt mapper som blir kopiert fra Github til egen datamaskin med kommandoet for kloning?
  8. Åpne en tekstredigeringsverktøy på egen datamaskin, åpne mappen hvor repository-en fra Github ble klonet (lagret i), velge filen README.md, skrive inn navn og versjon for operativsystemet på egen datamaskin på en ny linje og lagre endringene. Er det alltid nødvendig å åpne egen applikasjon for tekstredigering for å kunne gjøre endringer i filene? Kunne jeg endret andre type filer på samme måte, som, for eksempel, bilde- eller videofiler? Begrunne svar.
  9. Bruk egnede Git-kommandoer på kommandolinjen for å
    • sjekke status for endringene i den gjeldende mappen
    • registrere den endrede filen (ny versjon av filen og generelt endring i den gjeldende mappen) for innlegging i den lokale repository-en ("legge filen på scenen" eller på engelsk "add to staging area"),
    • forpliktende (på engelsk "commit") legge inn den nye versjonen av filen (endringene av filen og generelt i den gjeldende mappen) i repository-en,
    • kopiere eller overføre (på engelsk "push") endringene til en annen instans (som kan ha samme versjon som ved siste synkronisering eller en endret versjon) av den samme repository-en.
    Hvilken er gjeldende mappe på kommandolinjen? Hvorfor trenger man flere operasjoner for å registrere endringene i Git versjonskontrollsystemet? Kan man klone og kopiere endringene tilbake for en repository man ikke selv eier (dvs. repository-er som andre har laget ved hjelp av sine kontoer på Github)? Hvorfor er det nyttig å kopiere endringene gjort i en fil på egen datamaskin til Github?
  10. I nettleseren, i grensesnittet for github.com, og pålogget med din konto, gjør endringer i README.md filen. Legg inn brukernavnet for din github-konto på en ny linje i README-filen. Lagre filen i Github.
  11. Gå til tekstredigeringsverktøyet på din lokale datamaskin igjen og legg inn en ny linje i README.md med versjonsnummer til Git applikasjonen som du har installert på din egen datamaskin (for eksempel "git version 2.30.1 (Apple Git-130)". Lagre endringene. Prøv å registrere den endrede versjonen av README.md, legg inn endringene i den lokale versjonen av repository-en og sende/pushe endringene til Github. Hvilke respons genererer "push"-kommandoen? Hva er grunnen til en slik respons?
  12. Prøv å finne en løsning for å kunne ta vare både på de endringene som ble gjort i Github og de som ble gjort i den lokale repository-en.
  13. Konfigurer "Github pages" for din repository i Github fra grenen main og i mappen /docs. Hva er funksjonen til "Github pages"?
  14. I din lokale repository-en lag en mappe med navn docs og i denne mappen lag en fil med navn index.html. Hvorfor gjør man endringene i den lokale repository-en og ikke i Github direkte?
  15. I denne filen legg inn følgende kode: 
    <!DOCTYPE html>
<html>
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>(navn [mellomnavn]* etternavn)</title>
<</head>
<body>
  <h1>Min IS-114 portfolio</h1>
  <p>Kort beskrivelse av portfolio (oppgaveløsninger, dagbok, notater osv.) ... </p>
</body>
</html>
    Hvilken type kode er denne? Hvordan kan denne koden bli utført?
  16. Bruk git-kommandoer for å lagre endringene i Github. Hva betyr det at en webside er tilgjengelig på Internett?
  17. Gjør endringer (legg inn ditt navn i tittelen og lag en kort beskrivelse av din portfolio så langt) i filen index.html, lagre endringene lokalt og sende endringene til Github igjen. Hvorfor er en slik fremgangsmåte nyttig ved utvikling av websider?
  18. ...
  19. git merge feature-notater bruker 'ort' strategi for sammenslåing (Git - Merge-Strategies Documentation, 2023)
  20. git branch -d feature-notater for å fjerne grenen lokalt
  21. Må bruke git push origin -d feature-notater for å fjerne grenen fra Github.
  22. Hva hvis en utvikler sletter en feature-gren og en annen har den fortsett på sin lokale datamaskin?

Gruppedel

  1. Velg hvem av gruppemedlemmene, som skal eie HOVEDREPOSITORY.
  2. Eieren av HOVEDREPOSITORY lager en ny offentlig (ikke privat) repository med et navn som hele gruppen velger.
  3. Eieren av HOVEDREPOSITORY inviterer alle de andre gruppemedlemmene som samarbeidspartnere i denne repositorien.
  4. Bruk erfaring fra den individuelle delen og lag "Github pages" med index.html i mappen /docs i HOVEDREPOSITORY.
  5. Alle gruppemedlemmene kloner HOVEDREPOSITORY til sin lokale datamaskin.
  6. Alle gruppemedlemmene lager en gren med navn feature-(initialer) (hvis noen av gruppemedlemmene har like initialer, så må dere finne på en måte å skille mellom dem).
  7. I denne grenen endre index.html ved å legge inn et avsnitt med eget visitkort (kanskje lurt å unngå å legge ut mobilnummer, siden såkalte "crawlere" søker opp alle offentlig tilgengelige sider på WWW og "skraper" disse for informasjon som kan brukes i spamming/markedsføring).
  8. Legg inn en "Pull request" for din gren i HOVEDREPOSITORY.
  9. Alle sammen følger med på når eieren til HOVEDREPOSITORY godtar og legger inn alle endringene i grenen main.

Hva skal evalueres?

Vise deres læringsassitent den individuelle repository-en og websiden basert på den. Vise frem gruppe repository-en og websiden basert på den. Vise frem status for begge repository-ene på kommandolinje lokalt på sin datamaskin. Websidene må vises frem korrekt, dvs. ha gyldig HTML-kode. README.md i den lokale repositorien må inneholde 3 linjer (1: operativsystemets navn og versjon, 2: brukenavnet for Github-konto, 3: versjonsidentifikator til Git). "Github pages" må være konfigurert korrekt både i de individuelle og gruppe repository-en. HOVEDREPOSITORY må inneholde minst en "Pull request" fra hver av gruppemedlemmene. Bidragene fra gruppemedlemmene må være lagt inn i index.html filen og vises frem på websiden (gjennom "Github pages"). I tilfelle det har oppstått problemer med gjennomføring av noen av punktene i oppgaven, skal studenten eller gruppen forklare sin fremgangsmåte og beskrive problemet for LA-en. Dere må også være forberedt på å svare på hver av spørsmålene, som står i oppgaveteksten (LA-en vil velge hvilke spørsmål som blir stilt). Godkjent/Ikke godkjent, men det blir gitt flere forsøk på å få innleveringen godkjent.
